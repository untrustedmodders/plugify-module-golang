# Go Хост единого рантайма

В этой директории находится **хост единого рантайма** для модуля языка Go. Вместо того чтобы загружать каждый Go-плагин как C-библиотеку (стандартный режим), этот хост позволяет запускать все Go-плагины внутри **одного общего Go-рантайма** с использованием нативного пакета Go [`plugin`](https://pkg.go.dev/plugin).

## Как это работает

В стандартном режиме каждый плагин — это отдельный `c-shared` бинарник со своим Go-рантаймом, сборщиком мусора и пулом потоков. Хост единого рантайма меняет эту схему: сам хост загружается Plugify как C-библиотека, а все Go-плагины затем загружаются в этот хост как нативные Go-плагины (`.so`-файлы, собранные с `-buildmode=plugin`). Все они разделяют один экземпляр Go-рантайма.

```
Plugify Core
    └── libplugify-module-golang.so   (языковой модуль)
    └── libplugify-host-golang.so     (этот файл — загружается через LD_PRELOAD)
            ├── plugin_a.so           (Go-плагин, buildmode=plugin)
            ├── plugin_b.so           (Go-плагин, buildmode=plugin)
            └── plugin_c.so           (Go-плагин, buildmode=plugin)
```

## Преимущества

- Go-плагины могут **вызывать друг друга напрямую**, минуя слой маршалинга Plugify.
- Все плагины разделяют **один сборщик мусора** — меньше накладных расходов памяти и пауз GC.
- Выделяется **меньше потоков ОС** по сравнению с запуском N независимых Go-рантаймов.

## Ограничения и предупреждения

Этот режим использует пакет Go `plugin`, который имеет существенные оговорки. Внимательно ознакомьтесь с ними перед принятием решения об использовании.

> Источник: [официальная документация пакета Go `plugin`](https://pkg.go.dev/plugin#hdr-Warnings):

- **Ограниченная поддержка платформ.** Go-плагины поддерживаются только на **Linux, FreeBSD и macOS**. Это делает их непригодными для приложений, рассчитанных на переносимость.
- **Плохая поддержка детектора гонок.** Даже простые гонки данных могут не обнаруживаться автоматически. См. [golang.org/issue/24245](https://go.dev/issue/24245).
- **Более сложное развёртывание.** Все части программы должны находиться в правильных местах файловой системы во время выполнения.
- **Сложнее рассуждать о порядке инициализации.** Некоторые пакеты могут не инициализироваться вплоть до момента, когда приложение уже давно запущено.
- **Угроза безопасности.** Ошибки в коде загрузки плагинов могут быть использованы злоумышленником для загрузки опасных или ненадёжных библиотек.
- **Обязательная совместимость тулчейна.** Все плагины и хост **должны** быть скомпилированы с одной и той же версией Go, одинаковыми тегами сборки и соответствующими флагами и переменными окружения. Любое несоответствие приведёт к **краху во время выполнения**.
- **Идентичные зависимости обязательны.** Все общие зависимости хоста и плагинов должны быть собраны из абсолютно одинакового исходного кода.

> На практике это означает, что хост и все плагины должны собираться вместе одним человеком или одной системой сборки. Если это уже ваш случай, возможно, проще сгенерировать Go-файлы с blank-импортами всех плагинов и скомпилировать один статический исполняемый файл.

Для случаев, когда эти ограничения слишком строги, рассмотрите стандартный режим **C shared library** или традиционные механизмы межпроцессного взаимодействия (сокеты, RPC, разделяемая память).

## Сборка хоста

Рантайм-хост **не распространяется** в виде готового бинарника — вы должны собрать его самостоятельно, чтобы он точно совпадал с тулчейном и зависимостями ваших плагинов.

Из этой директории:

```sh
go build -buildmode=c-shared -o libplugify-host-golang.so ./
```

## Развёртывание

Поместите полученный `libplugify-host-golang.so` в ту же папку **`bin`**, что и бинарник языкового модуля:

```
bin/
├── libplugify-module-golang.so   ← языковой модуль
└── libplugify-host-golang.so     ← этот файл
```

### LD_PRELOAD (обязательно)

Библиотека хоста **должна** быть предзагружена до старта Plugify. Если она инициализируется слишком поздно, Go-рантайм внутри хоста может оказаться не готов к моменту загрузки плагинов, что приведёт к сбою.

Установите `LD_PRELOAD` перед запуском Plugify:

```sh
LD_PRELOAD=/path/to/bin/libplugify-host-golang.so ./plugify
```

Или экспортируйте переменную в вашей оболочке / юните сервиса:

```sh
export LD_PRELOAD=/path/to/bin/libplugify-host-golang.so
```

## Сборка плагинов для режима единого рантайма

Плагины, предназначенные для хоста единого рантайма, собираются иначе, чем в стандартном режиме C shared library.

Используйте `-buildmode=plugin` и тег сборки `-tags=plugin`, чтобы генерируемые биндинги были скомпилированы корректно:

```sh
go build -buildmode=plugin -tags=plugin -o my_plugin.so ./
```

> **Важно:** Все плагины и хост должны использовать одну версию Go, одинаковые зависимости модулей (зафиксированных на идентичных версиях и исходниках) и одинаковые теги сборки. Смешивание не поддерживается и приведёт к сбою.

## Выбор подходящего режима

| | C Shared (стандартный) | Единый рантайм (этот) |
|---|---|---|
| Поддержка платформ | Все платформы | Только Linux, FreeBSD, macOS |
| Вызовы между плагинами | Через маршалинг Plugify | Прямые Go-вызовы |
| GC / потоки | Отдельные для каждого плагина | Общие для всех плагинов |
| Сложность сборки | Низкая | Высокая — всё должно совпадать |
| Переносимость | Высокая | Низкая |
| Рекомендуется для | Общего использования | Критичных по производительности, контролируемых окружений |
